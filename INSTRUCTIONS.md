# Working instructions for this repo

Read this before making changes. For "what happened and why," see `HANDOFF.md`.
For "what's live right now," see `STATUS.md`.

## The owner is non-technical

The entire point of `/admin` is that Mujahid (the site owner) edits content —
text, prices, photos — without touching code or asking a developer. Any change
to `src/lib/site-content.ts`'s schema, or to what fields the admin panel
exposes, should be weighed against: **can he still make routine edits himself
after this change?**

## Hard rules — do not violate these

1. **Never write a secret value into a file that gets committed.** Not into
   `STATUS.md`, not into a commit message, not into a code comment. Real
   values (admin password, SMTP credentials, KV namespace IDs if ever used in
   `wrangler.toml`) live in: local `.env` (gitignored) and Cloudflare Pages'
   dashboard secrets. If you need the current admin password, ask the user —
   don't paste it into a doc "for reference."

2. **Never add a `node:fs` read/write for anything that needs to work on
   Cloudflare.** Cloudflare Workers have no filesystem at runtime. This bit us
   once already (see HANDOFF.md Round 3) and would bite again for any new
   static data file. If you need build-time-known JSON/data available at
   runtime, use a static ES `import` so it's bundled into the JS, not a
   dynamic `readFile` call.

3. **Never read Cloudflare secrets/bindings via `process.env` or
   `vinxi/http`'s `getEvent()`.** Neither reliably works on this app's runtime
   (nitro, not vinxi; Cloudflare Pages doesn't populate `process.env` from
   dashboard secrets). Use `getCloudflareEnv()` / `getEnvVar()` from
   `src/lib/cloudflare-env.ts` — those read from `globalThis.__CF_ENV__`,
   which `dist/client/_worker.js` (generated by `scripts/cf-postbuild.mjs`)
   stashes at the top of every request. If you add a new Cloudflare binding
   (e.g. an R2 bucket), wire it through this same path.

4. **Don't touch DNS records for MX, SPF, DKIM, or DMARC** on
   procleanorganizers.com without explicit confirmation from the user. Those
   run the owner's actual business email through Hostinger. They must stay set
   to "DNS only" (grey cloud) in Cloudflare — never "Proxied" (orange cloud).
   Verified safe as of 2026-07-12; don't assume that's still true without
   checking if significant time has passed or someone else touched DNS.

5. **Don't connect the custom domain** (procleanorganizers.com) to the
   Cloudflare Pages project unless the user explicitly says to. As of this
   writing they said "not ready to connect the domain yet" — that instruction
   holds until they say otherwise, not just for one session.

6. **Verify against the live URL after any deploy-affecting change**, don't
   just trust that Cloudflare's build succeeded. A build can succeed (files
   uploaded fine) while the Worker still throws at runtime — this happened
   three separate times in this project (see HANDOFF.md). Curl the API routes
   directly to check status codes; use the browser tools to confirm the full
   page/login flow actually works, not just that `git push` succeeded.

## Build & deploy mechanics

- **Local dev:** `bun run dev` (or the `dev` launch config in `.claude/launch.json`)
- **Build:** `bun run build` — this runs `vite build` *and* `node scripts/cf-postbuild.mjs`.
  Don't run `vite build` alone and expect a working Cloudflare deploy; the
  postbuild step is what generates `_worker.js`.
- **Deploy:** automatic — any push to `main` on GitHub triggers a Cloudflare
  Pages build+deploy. There is no manual deploy step.
- **Package manager:** bun. Repo has `bun.lock`; don't introduce `package-lock.json`
  or `yarn.lock`.

## Git push in this environment

`git push` from a foreground shell command reliably hangs — Git Credential
Manager needs to show a native Windows popup for auth, and a sandboxed
foreground command can't surface it. Push with `run_in_background: true`,
tell the user to approve the popup on their desktop, wait for the background
task's completion notification, then confirm with
`git fetch origin main -q && git log origin/main --oneline -1`.

## Content schema changes

`src/lib/site-content.ts` exports `siteContentSchema` (Zod). If you add/remove
a field:
- Update `data/site-content.seed.json` to match (this is the bundled fallback —
  see rule #2 above, it's a static import, not a runtime file read)
- Update the admin form in `src/routes/admin.tsx` so the owner can actually
  edit the new field
- Existing KV-stored content (once KV is bound) won't automatically have new
  fields — `siteContentSchema.parse()` will throw on old data if you add a
  required field without a `.default(...)`. Prefer adding new fields with
  Zod defaults so old saved content doesn't break.

## When in doubt

Check `STATUS.md` for current live state, `HANDOFF.md` for why things are the
way they are, and ask the user before touching DNS, the custom domain, or
anything that sends real email/money/messages to real people.
